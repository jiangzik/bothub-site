# 供应商与模型配置

## 核心字段

| 字段 | 用途 | 常见错误 |
| --- | --- | --- |
| `基础地址` (Base URL) | 供应商接口地址 | 写错协议或多了路径 |
| `API Key` | 鉴权凭证 | 复制不完整、已过期 |
| `API 协议` | 接口协议类型 | 选错协议导致请求格式不对 |
| `Model ID` | 要调用的模型 | 拼写错误 |

## API 协议

添加厂商时需要选择 API 协议：

- **OpenAI** — 适用于 OpenAI 及兼容 OpenAI 格式的服务商（大部分国内外厂商都兼容）
- **Anthropic** — Claude 系列
- **Google** — Gemini 系列，需要额外填项目 ID 和区域
- **其他** — 按实际情况选择

OpenAI 协议还可以选标准模式 (Chat Completions) 或新版模式 (Responses API)。

## 高级选项

- `自定义请求头 (JSON)` — 有些供应商需要额外的 header
- `自定义请求体 (JSON)` — 需要在请求体中加额外字段时使用

## 建议的配置方式

1. 至少配两个模型（主力 + 备用）。
2. 同一厂商下的不同模型分开保存，方便切换。
3. 按用途命名，比如"写作-长文""代码-快速"。

## 出了问题按这个顺序查

1. API Key 是否有效（去供应商后台看一眼）
2. Model ID 是否和供应商后台一致（直接复制）
3. Base URL 是否多了路径或写错了协议
4. 是不是网络代理或防火墙拦了请求

## 参数建议

- **Temperature**：问事实调低，写创意调高
- **Max Tokens**：太小会截断回复，太大可能增加费用和延迟
- **Top P**：一般保持默认，和 Temperature 不要同时调

## 安全建议

- 不要把 API Key 提交到公开仓库
- 定期换 Key
- 发现泄露了立刻在供应商后台禁用旧 Key
